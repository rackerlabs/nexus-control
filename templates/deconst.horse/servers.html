List Servers #

You can list the servers in a particular region in summary form, with only the name and ID returned, or in detailed form, with all available fields returned.

API Request #

To retrieve a summary list of servers in a particular region, send a GET HTTP request to /v2/{tenantId}/servers, replacing {tenantId} with your tenant ID.

To retrieve a detailed list of servers in a particular region, send a GET HTTP request to /v2/{tenantId}/servers/detail, replacing {tenantId} with your tenant ID.

Various filters may be applied to either the summary or detailed lists by appending any of the following query string parameters to the request:

Filter Parameters
Parameter Key Description
image Only list servers with this image ID. For a list of images, see Section 2.7.1, “List Images”.
flavor Only list servers with this flavor ID. For a list of flavors, see Section 2.6.1, “List Flavors”.
name Only list servers with this name.
status Only list servers with this status. See Server Status Values.
marker The ID of the last item in the previous page of entries. See Section 1.6, “Paginated Collections”.
limit The number of servers to show in a single page.
changes-since Only show servers that have changed since this date/time. See Section 1.7, “Efficient Polling with the Changes-Since Parameter”.
RAX-SI:image_schedule Whether scheduled images are enabled or not. If the value is the boolean keyword TRUE, the list contains all servers that have an image schedule resource set on them. If the value is the boolean keyword FALSE, the list contains all servers that do not have an image schedule. (This URI parameter is added by the Rackspace Scheduled Images API Extension. See Section 3.5, “Scheduled Images Extension”).

API Request

Request Headers: Summary
GET /v2/123456/servers
X-Auth-Token: your_auth_token
Accept: application/json
Request Headers: Detail
GET /v2/123456/servers/detail
X-Auth-Token: your_auth_token
Accept: application/json
Request Headers: with Filters
GET /v2/123456/servers/detail?status=SUSPENDED
X-Auth-Token: your_auth_token
Accept: application/json

Servers may have a variety of statuses during their lifecycle:

Server Statuses
Status Description
ACTIVE The server is active and ready to use.
BUILD The server is being built.
DELETED The server was deleted. The list servers API operation does not show servers with a status of DELETED. To list deleted servers, use the changes-since parameter. See Section 1.7, “Efficient Polling with the Changes-Since Parameter”.
ERROR The requested operation failed and the server is in an error state.
HARD_REBOOT The server is going through a hard reboot. A hard reboot power cycles your server, which performs an immediate shutdown and restart. See Section 2.4.2, “Reboot Server”.
MIGRATING The server is being moved from one physical node to another physical node. Server migration is a Rackspace extension.
PASSWORD The password for the server is being changed. See Section 2.4.1, “Change Administrator Password”.
REBOOT The server is going through a soft reboot. During a soft reboot, the operating system is signaled to restart, which allows for a graceful shutdown and restart of all processes. See Section 2.4.2, “Reboot Server”.
REBUILD The server is being rebuilt from an image. See Section 2.4.3, “Rebuild Server”.
RESCUE The server is in rescue mode. Rescue mode is a Rackspace extension. See Section 3.3, “Rescue Mode Extension”.
RESIZE The server is being resized and is inactive until the resize operation completes. See Section 2.4.4, “Resize Server”.
REVERT_RESIZE A resized or migrated server is being reverted to its previous size. The destination server is being cleaned up and the original source server is restarting. For a server that was resized, see Section 2.4.4, “Resize Server”. Server migration is a Rackspace extension.
SUSPENDED The server is inactive, either by request or necessity. Review support tickets or contact Rackspace support to determine why the server is in this state.
UNKNOWN The server is in an unknown state. Contact Rackspace support.
VERIFY_RESIZE The server is waiting for the resize operation to be confirmed so that the original server can be removed.
Error Response Codes
Code Description
400 Bad Request. Typically caused by a parameter of the wrong type.
401 Unauthorized. Typically caused by an invalid authentication token or tenant ID.
403 Forbidden.
405 Method Not Allowed. Caused by sending a request with an invalid HTTP verb. For listing servers, GET is the only valid request type.
413 Over Limit. You have exceeded your limit for API requests of this type.
503 Service Unavailable. An internal error is is preventing the API from serving requests.

Create a Server #

This operation asynchronously provisions a new server. After submitting a JSON document with configuration details for the new server, the API will provide a progress URL that can be polled to monitor the server's creation status.

API Request #

To create a server, send a POST HTTP request to /v2/{tenantId}/servers, where {tenantId} is the tenant ID for which you would like to create the server.

Your request body should be a JSON document with a single object named server, containing several properties that will determine what kind of server you will create.

API Request Fields
Parameter Type Description
name string The name of your server
imageRef string The id of the image you want to use when creating your server. This is often referred to as the operating system.
flavorRef string The id of the flavor you want to use when creating your server. This is often referred to as the size of the instance.
key_name
optional 		string The name of the SSH Key to use when creating your server. The name must already exist. See creating SSH keys for more information.
key_name is required if the flavorRef is an OnMetal flavor.
metadata
optional 		object Key/Value pairs for metadata to attach to your server.
networks
optional 		array One or more object with the key uuid and value of the network id to attach to this instance. See using networks when creating a server for more information.
personality
optional 		array One or more object with the path of the file to create and contents (Base64 encoded) of the file contents. See using personality when creating a server for more information.
user_data
optional 		string The base64 contents of a file to be created.
OS-DCF:diskConfig
optional 		string Used to set the disk configuration to MANUAL. Default is set to AUTO. See more details about disk configuration.
config_drive
optional 		boolean If set to true, the system will use data in one or both of user_data and personality to configure the server. Defaults to false.

If an image has OS-DCF:diskConfig value of MANUAL, you cannot create a server from that image with a OS-DCF:diskConfig value of AUTO.

Important

OnMetal servers must be created using an ssh key pair. Thus you should ignore the administrator password returned by a Create Server operation because it does not allow access to the OnMetal server. See Section 2.2, “Server Key Pairs” for more information about key pairs.

API Response #

Although the API will respond immediately with some information, the process of actually creating the server will take some time. The initial response from the API gives you all the information you need in order to check on the server's status while it's being created.

The full URL to the newly created server is returned through the Location header and is available as a self and bookmark link in the server representation. See Links and References.

When you create a server, only the server ID, its links, and the administrator password are guaranteed to be returned in the request.

Status Codes

The API will respond with the status code 202 Accepted if the creation request was successful. Any other status code indicates an error.

Error Response Codes
Status Code Reason
400 Bad Request. Often as a result of invalid arguments.
401 Unauthorized. Check your Auth-Token. See Authentication for more on Authentication Tokens.
403 Forbidden.
404 Item not found.
405 Method not allowed.
413 Over limit. You may need to request increased limits for your account. See limits for more on resource limits.
415 Bad media type.
500 Compute fault.
503 Service unavailable.
503 Server capacity unavailable. This is an exceptional error that may only occur when region capacity has been exhausted.

Checking Progress #

After a successful request to create a server, the API will provide a URL that can be used to monitor the server creation process. You can check the progress of your server build by sending a GET request to the new server's URI.

The progress property will be an integer between 0 and 100, indicating the completion percentage.

The status property will be "BUILD" until the build has either succeeded or failed. If the build was successful, this property will change to "ACTIVE". If the build failed, this property will change to "ERROR"

Several other properties of the server object may not be populated until the server has completed building.

Tip

For OnMetal server builds, even after the status is ACTIVE, wait a few additional moments for the network configuration to complete. Once the new OnMetal server pings successfully, you may begin to use it.
